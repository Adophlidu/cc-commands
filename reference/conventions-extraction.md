# conventions-extraction

Instructs `/d:init` how to produce `docs/conventions.md` in the target project — the authoritative code-style and pattern reference for all `d` agents.

## Output Location

`docs/conventions.md` in the target project root.

---

## Overview

`docs/conventions.md` is a **tight reference**, not a style guide narrative. It records the current truth about how this project is written so every `d` agent writes consistent code. Produce it by merging three sources in order:

1. **Explicit** — what config files declare.
2. **Inferred** — what the code actually does, grep-verified.
3. **Best practice (context7)** — what the framework's current official docs recommend, folded in only where the project hasn't already decided.

---

## Step 1 — Collect Explicit Conventions

Scan the target project for each of the following. Record what you find; note `absent` for anything missing.

| Config file | What to extract |
|---|---|
| `.eslintrc*` / `eslint.config.*` | Rule set, extends, plugins, any overrides |
| `biome.json` | Formatter/linter settings (indent, quotes, semicolons, organizeImports) |
| `.prettierrc*` / `prettier.config.*` | printWidth, tabWidth, singleQuote, semi, trailingComma |
| `tsconfig.json` | `strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, target, module resolution |
| `.editorconfig` | indent_style, indent_size, end_of_line, charset |
| `CONTRIBUTING.md` | Any explicit code-style sections (branch naming, commit format, PR rules) |
| `CLAUDE.md` (project-level) | Any agent-facing conventions already recorded |

### Hard rule: author missing config

If **lint, format, or typecheck config is entirely absent**, author a minimal config using the framework's recommended preset before proceeding. Examples:

- TypeScript project with no eslint config → create `eslint.config.js` using `eslint:recommended` + `typescript-eslint/recommended`.
- No prettier/biome → create `.prettierrc` with the framework's canonical defaults.
- No `tsconfig.json` → create one with `"strict": true` and the framework's recommended `target`/`moduleResolution`.

This is required so the `d-reviewer` quality gate has something to run. Record the authored config in `docs/conventions.md` under a clearly marked **"Authored by /d:init"** note.

## Step 2 — Infer Conventions from Code

Read a representative sample of source files (aim for ≥10 files spanning different layers). Look for:

- Naming: variable/function/class/file naming style (camelCase, PascalCase, kebab-case, snake_case).
- Imports: barrel files, path aliases, absolute vs relative, import ordering.
- Error handling: thrown errors vs returned Result types, custom error classes, try/catch placement.
- Async patterns: async/await vs promise chains, error propagation.
- Module structure: how files export (named vs default exports), index re-exports.
- Test structure: describe/it nesting depth, fixture patterns, mock style.
- Common pitfalls: any patterns that appear broken, duplicated, or inconsistently applied.

### Hard rule: grep-verify before codifying

**Every inferred convention must be confirmed to actually dominate the repo before being written into `docs/conventions.md`.** Use `grep` (or Bash `grep -r`) across the repo to count occurrences:

```bash
# Example: confirm camelCase function naming dominates
grep -rE "^(export )?function [a-z][a-zA-Z]+" src/ --include="*.ts" | wc -l
grep -rE "^(export )?function [A-Z][a-zA-Z]+" src/ --include="*.ts" | wc -l
```

If the counts are close (within 20 %), mark the convention as **inconsistent** in `docs/conventions.md` rather than picking a winner. Do not impose a pattern the project doesn't follow.

## Step 3 — Fold in Best Practices via context7

Use the context7 MCP to pull the chosen framework's current official style and best-practice guide. The context7 MCP is available via `ToolSearch` — resolve the library ID first, then query docs:

```
ToolSearch: select:mcp__plugin_context7_context7__resolve-library-id
ToolSearch: select:mcp__plugin_context7_context7__query-docs
```

Steps:
1. Resolve the library ID for the primary framework detected in Step 1 (e.g. `react`, `nextjs`, `hono`, `fastapi`).
2. Query for "code style", "project structure", and "error handling" topics.
3. For each best practice returned: **only add it to `docs/conventions.md` if the project has not already made a decision** (explicit or inferred). Never override an existing project convention with a framework default.

Label any context7-sourced entry with `(framework default)` so agents know it is advisory, not project-specific.

## Step 4 — Write `docs/conventions.md`

Produce a single, scannable file. Target: readable in under 3 minutes. Use the following section structure:

### Required sections

```markdown
# Conventions

> Generated by /d:init. Update via `/d:init --refresh` or edit manually.

## Naming
<!-- Identifiers, files, directories, branches, DB columns -->

## Structure & Layout
<!-- Directory conventions, module boundaries, barrel/index files, import ordering -->

## TypeScript / Language
<!-- Strictness flags in effect, type patterns to prefer/avoid -->

## Error Handling
<!-- Thrown errors vs Result types, custom error classes, async error propagation -->

## Testing
<!-- Framework, file placement, describe/it conventions, fixture/mock patterns -->

## Formatting & Linting
<!-- Active tools and key settings (inferred from config files) -->

## Known Pitfalls
<!-- Patterns observed in the repo that cause bugs or are being phased out -->

## Authored Config (if any)
<!-- Configs created by /d:init because none existed. Remove this section once the team adopts them. -->
```

### Lean rule

- Each section is bullet points or a short table — no prose paragraphs.
- Record only what is **true now** in the project. No aspirational rules.
- Cite the source of each entry: `(eslint)`, `(tsconfig)`, `(inferred, grep-verified)`, `(framework default)`, or `(CONTRIBUTING)`.
- Do not reproduce source code. Reference file paths only when a specific file is the canonical example.

## Verification Checklist

Before finalizing `docs/conventions.md`, confirm:

- [ ] Every config file listed in Step 1 was checked (or marked `absent`).
- [ ] Every inferred convention was grep-verified with a count > the alternative.
- [ ] context7 was queried for the primary framework and results were reviewed.
- [ ] Missing lint/format/typecheck config was authored (if applicable).
- [ ] The file fits the required section structure above.
- [ ] No section contains uncited claims.
